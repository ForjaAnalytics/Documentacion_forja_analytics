07 · Integraciones

Cómo habla Emerald con cada plataforma

Tres integraciones, tres mecanismos distintos. La elección no fue arbitraria: vino impuesta por lo que cada plataforma expone.

JazzHR

API REST · solo lectura. shared/jazzhr/client.py rechaza cualquier método ≠ GET cuando JAZZHR_READ_ONLY=true — defensa para no mutar jamás el ATS de producción del cliente.

Endpoints: /jobs, /jobs/status/open, /applicants (global), /applicants/job_id/{id}, /applicants/{id}.

Un bot Playwright secundario (web_bot.py) descarga CVs cuando el S3 público devuelve 403.

HiPeople

Solo bot Playwright. No hay API de escritura usable y la de lectura está bloqueada por plan en este tenant. Toda interacción es UI automatizada.

Sesión en hipeople_storage_state.json, replicada como env var base64 para el deploy. Un login (con MFA si aplica) se reutiliza indefinidamente.

Es el punto más acoplado a la UI: si HiPeople refactoriza su interfaz, hay que actualizar selectores.

RecruiterFlow

API REST. shared/recruiterflow/client.py, auth por header RF-Api-Key. Es el destino de escritura: mueve y descalifica candidatos.

Endpoints: /candidate/search, /add, /add-to-job, /move-to-stage, /disqualify, /file/add, /job/list, /job/{id}/pipeline.

Deduplica por email (HTTP 409 → recupera el id existente).

Sesiones de navegador. Las sesiones Playwright se persisten como archivos JSON en data/ y se replican en las env vars EMERALD_HIPEOPLE_STORAGE_STATE_B64 y EMERALD_JAZZHR_STORAGE_STATE_B64 (base64). Un contenedor de bootstrap materializa el archivo desde la env var al arrancar — así el deploy en Railway no necesita un volumen persistente para las cookies.
