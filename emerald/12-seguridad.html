Autenticación & superficies sensibles

El dashboard tiene auth propia; las integraciones se protegen por diseño defensivo.

Auth del dashboard

bcrypt (cost 12) + JWT HS256 en cookie httpOnly emerald_session (TTL 24h). Roles owner / admin / viewer. El frontend nunca lee el token.

JazzHR read-only

El cliente rechaza todo método ≠ GET cuando JAZZHR_READ_ONLY=true — imposible mutar accidentalmente el ATS de producción del cliente.

Autorización de workers

Vive en la DB, no en env vars: un worker solo actúa sobre grupos con creation_decision='approved' + sync_status='synced'. El admin aprueba desde el dashboard.

Secretos del entorno. El archivo .env contiene las credenciales reales (API keys de JazzHR / HiPeople / RecruiterFlow, passwords, JWT secret) y data/*.json contiene las cookies de sesión activas. Ambos están excluidos del repositorio mediante .gitignore y existen únicamente en el filesystem local del entorno de ejecución; se tratan como secretos y no se incluyen al compartir o respaldar el proyecto.

Env vars sin uso. Por la evolución del sistema en fases, hay variables que siguen presentes en Railway pero que el código ignora: EMERALD_RF_DRY_RUN, EMERALD_*_ALLOWED_GROUPS, EMERALD_BOT_ALLOW_PRODUCTION, EMERALD_OPEN_GROUPS, EMERALD_TEST_USER_EMAILS. La lógica real de autorización es la decisión por grupo almacenada en la base de datos.
