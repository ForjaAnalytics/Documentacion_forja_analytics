Alta de grupos & runbook

Lo necesario para operar el sistema. El detalle exhaustivo vive en CLAUDE.md y ARCHITECTURE.md.

Alta de un grupo nuevo

Automático
El poller descubre la vacante → crea filas en job_group + job_mapping con creation_decision='pending_decision'.
Humano · dashboard
Aprobar el grupo eligiendo el flow (/api/decisions/{key}/approve).
Automático · sender
Crea el rol en HiPeople en el siguiente ciclo (~10s).
Humano · HiPeople UI
Configurar módulos (Interview with AI + criteria + assessment template).
Automático
Auto-detección de readiness marca assessment_ready_at en ≤60s → empiezan las invitaciones.

Síntomas y causas

SíntomaCausa / acción
Sender no invita aunque el grupo está approved+syncedassessment_ready_at IS NULL. Configurar el rol en HiPeople o llamar mark-template-ready.
Promoter reporta "0/N completed with score"HiPeople usa un badge de estado no reconocido. Ampliar el matcher en scrape_interview_results.
Candidato "se mueve hacia atrás" en RFBug de sibling: dos prospectos con el mismo email. derive_target retorna noop para skipped+duplicado.
Stages renombrados en RF no se respetanEl cache stages_json no expira. Invalidar: UPDATE rf_job_mapping SET stages_json='{}'.
Bot crashea creando el rolHiPeople cambió selectores del wizard. Revisar screenshots en data/screenshots/create_role/ y actualizar hipeople_bot.py.
Sesión HiPeople/JazzHR expiradaRe-bootstrap login, actualizar env var *_STORAGE_STATE_B64, redeploy.

Lag de JazzHR

  • El endpoint global /applicants tarda ~30–60 s en reflejar un applicant nuevo → por eso el poller corre cada 5 min.
  • El endpoint per-job /applicants/job_id/{id} tiene lag de varios minutos → por eso la pasada global es la primaria y la per-job solo cubre el bug de omisión.
  • POST /jobs de JazzHR tiene validaciones crípticas → las vacantes se crean manualmente en la UI de JazzHR.
