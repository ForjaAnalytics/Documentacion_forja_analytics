Flujo de una consulta, de extremo a extremo

Dónde vive cada dato y qué componente lo mueve, paso a paso:

navegador → Drive
1 · Captura y subida
El SPA graba el audio y pide POST /api/uploads/init. El backend abre una sesión resumable y devuelve una session_uri; el navegador hace el PUT del blob directo a Drive. El audio nunca atraviesa el backend.
navegador → API → DB
2 · Creación de la consulta
Con el drive_file_id, el SPA llama POST /api/consultations. El backend persiste la fila en queued, responde de inmediato y agenda pipeline.run() como BackgroundTask.
pipeline (async)
3 · Transcripción
El pipeline baja el audio (drive.download_file), lo normaliza con ffmpeg y lo envía a ElevenLabs Scribe. El transcript se guarda en la fila (estado transcribing).
pipeline (async)
4 · Extracción
El transcript se reescribe como diálogo etiquetado y se pasa a Claude con un schema construido desde el capture_config del usuario. La extraction y las métricas de uso se guardan; el estado pasa a ready (o error).
navegador ← API
5 · Revisión
La pantalla de resultados hace polling de GET /api/consultations/:id hasta ready, muestra los campos editables y guarda los ajustes del médico con PATCH. El export a Word ocurre en el navegador.

Principio · el audio no pasa por el backend

La subida es resumable y directa a Drive (patrón tipo presigned URL): el backend solo abre la sesión. Esto evita un cuello de botella de ancho de banda y mantiene la API liviana — solo maneja JSON.

Principio · trabajo pesado en segundo plano

Transcribir y extraer toma segundos o minutos. Corren como BackgroundTask tras responder al cliente, y el avance se comunica por el campo status que el frontend consulta por polling. La consulta es la única fuente de verdad del progreso.
