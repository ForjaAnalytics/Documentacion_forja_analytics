06 · El sistema

Captura configurable

El capture_config es un JSON que define qué campos extrae la IA. Cambiarlo cambia lo que Claude busca —sin tocar código— porque el schema de la herramienta se construye dinámicamente a partir de él.

Cada usuario tiene su propio capture_config (columna JSONB en users), editable vía PUT /api/me/capture-config. Al registrarse, se copia desde el capture_config_default del account, que a su vez parte del DEFAULT_CAPTURE_CONFIG del backend. Un campo se describe así:

{
  "name": "paciente_edad",
  "type": "number",
  "label": "Edad",
  "required": true,
  "description": "Edad del paciente en años. null si no se menciona."
}

La description es lo más importante: es el texto que Claude lee para decidir qué poner en el campo. Buenas descripciones → buenas extracciones.

Tipos de campo soportados

TipoJSON Schema generadoUso típico
string / text["string","null"]Nombre, motivo, texto libre.
number["number","null"]Edad, temperatura, número de hijos.
boolean["boolean","null"]Indicadores sí/no.
date["string","null"] + format:dateFechas.
enum["string","null"] + enumTipo de atención (control, urgencia…).
object["object","null"] con childrenSecciones agrupadas (signos vitales, antecedentes).
arrayarray de primitivos u objetosSíntomas, medicamentos, diagnósticos.

De config a herramienta de Claude

extract.build_tool recorre los campos y construye el input_schema de una herramienta llamada extract_consultation. Luego se fuerza a Claude a usarla con tool_choice, así que devuelve un objeto ya estructurado en tool_use.input —nada de parsear JSON de texto plano.

tool = build_tool(capture_config)
tool["cache_control"] = {"type": "ephemeral"}   # schema cacheado 5 min

resp = client.messages.create(
    model=settings.anthropic_model,
    system=SYSTEM,
    tools=[tool],
    tool_choice={"type": "tool", "name": "extract_consultation"},
    messages=[{"role": "user", "content": transcript_text}],
)

El schema se marca con cache_control: ephemeral (prompt caching): consultas consecutivas dentro de ~5 minutos no vuelven a pagar los tokens del schema, que es la parte grande y estable del prompt.

El esquema base de cuidados paliativos

El DEFAULT_CAPTURE_CONFIG fue curado por la Dra. María Alejandra Umbacía y cubre la valoración integral. Sus secciones:

General

Datos demográficos y de contexto: nombre, edad, lugar de nacimiento y de vivienda, ocupación, escolaridad, con quién vive, número de hijos, teléfono, dirección, correo, acompañante en la consulta.

Motivo de consulta

El motivo principal por el que se valora al paciente.

Enfermedad actual

Descripción cronológica del problema de salud: inicio, evolución, características, factores y tratamientos recibidos.

Revisión por sistemas

Interrogatorio sistemático con cada síntoma calificado 0–10: dolor, fatiga, náuseas, tristeza, ansiedad, somnolencia, disnea, apetito, bienestar, sueño y más.

Antecedentes

Patológicos, farmacológicos, quirúrgicos, ginecológicos (GPCA), familiares, alérgicos y toxicológicos.

Signos vitales

Presión arterial y temperatura tomadas o mencionadas.

Síntomas e impresión diagnóstica

Lista de molestias del episodio, y los diagnósticos que el médico enuncia explícitamente.

Plan, tipo y notas

Medicamentos prescritos (nombre, dosis, frecuencia), tipo de atención (enum) y notas libres para lo que no encaje en otro campo.

Regla de oro: no inferir. El campo de impresion_diagnostica instruye a Claude a recoger solo los diagnósticos que el médico enuncia explícitamente en el audio. Tiene prohibido inferir un diagnóstico a partir de los síntomas o los antecedentes. El SYSTEM prompt también recuerda no inventar datos: si un campo no aparece, va null (o [] para arrays). Esto mantiene la herramienta como ayudante de documentación, no de diagnóstico.
