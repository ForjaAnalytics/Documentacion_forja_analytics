12 · Infraestructura

Seguridad

El sistema maneja datos clínicos sensibles. Estas son las superficies de seguridad y cómo se protege cada una.

Autenticación — JWT

Tokens HS256 firmados con JWT_SECRET, con claim sub = user_id y expiración de JWT_EXPIRE_DAYS (30 por defecto). El frontend los envía en Authorization: Bearer. get_current_user los decodifica y distingue token expirado de inválido (ambos → 401).

Contraseñas — bcrypt

Se almacenan solo hashes bcrypt (hash_password / verify_password), con salt por contraseña. El cambio de contraseña exige enviar la actual + la nueva juntas y valida la actual antes de rotar.

Autorización — roles por dependencia

El control de acceso se aplica con dependencias de FastAPI: require_super_admin, require_account_owner (owner o super_admin) y get_current_account, que bloquea (403) si el account está suspended/cancelled —salvo para super_admin.

Aislamiento por tenant

Las consultas de queries filtran siempre por user_id (y por account_id en vistas de account). Un doctor no puede leer pacientes ni consultas de otro: los GET /:id devuelven 404 si el recurso no es suyo.

Cuentas suspendidas

Un user suspended no puede iniciar sesión ni operar (403). El owner puede suspender miembros, pero no a sí mismo ni al owner del account.

Impersonación auditable

El super_admin obtiene un token corto (1 h) con el claim impersonated_by, que permite al frontend mostrar un banner y deja rastro para auditoría. No se puede impersonar a otro super_admin.

Minimización de datos

El overview del account (business) expone solo metadata —nombre del paciente, doctor, conteo y fecha— y omite documento, edad, sexo, contacto, notas, transcript, extracción y drive_file_id.

Secretos y CORS

El JSON de la service account vive fuera del repositorio (secrets/ en .gitignore, o como variable de entorno en producción). El CORS está restringido a los orígenes de CORS_ORIGINS. Las claves de IA y el JWT_SECRET nunca se hardcodean.

Sesión en el navegador

El token y el perfil se guardan en localStorage. El cliente de API reacciona a un 401 borrando el token y emitiendo cv:unauthorized, que el store usa para cerrar la sesión y limpiar el estado (incluida la revocación de URLs de blob de audio). Esto evita dejar sesiones colgadas con tokens caducos.

Pendientes / consideraciones

El billing con Stripe está como placeholder. Para un despliegue clínico real conviene revisar: cifrado/retención de los audios en Drive, registro de auditoría persistente y acuerdos de tratamiento de datos según la normativa aplicable.
