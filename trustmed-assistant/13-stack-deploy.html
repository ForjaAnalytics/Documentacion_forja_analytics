Stack & Deploy

Backend y frontend se despliegan por separado: la API en Railway, el SPA en Vercel. Cada uno con su archivo de configuración en el repo.

Backend Railway (NIXPACKS)

railway.json define el build con NIXPACKS y el deploy: arranca con uvicorn main:app --host 0.0.0.0 --port $PORT, hace healthcheck en /health y reinicia ante fallos. Existe un Procfile equivalente para entornos tipo Heroku. nixpacks.toml añade ffmpeg al sistema (lo necesita la normalización de audio).

// railway.json
{
  "build": { "builder": "NIXPACKS" },
  "deploy": {
    "startCommand": "uvicorn main:app --host 0.0.0.0 --port $PORT",
    "healthcheckPath": "/health",
    "restartPolicyType": "ON_FAILURE"
  }
}

En el arranque, el lifespan de FastAPI: (1) crea/promueve al super admin si SUPER_ADMIN_* está configurado, y (2) valida que la service account pueda leer la carpeta de Drive, registrando un error claro si no. Las migraciones se aplican con alembic upgrade head antes de levantar la API.

Frontend Vercel (estático)

Build estático con Vite (vite build). vercel.json reescribe todas las rutas a /index.html para que el routing del SPA funcione con recargas y deep-links. La URL del backend se inyecta en build con VITE_API_URL (por defecto http://localhost:8000).

// vercel.json
{ "rewrites": [ { "source": "/(.*)", "destination": "/index.html" } ] }

Dependencias clave

Backend (Python)

fastapi · uvicorn[standard] · pydantic + pydantic-settings

sqlalchemy · psycopg[binary] · alembic

google-auth · httpx · requests

anthropic · PyJWT · bcrypt

Frontend (JavaScript)

react 18 · react-dom · react-router-dom 6

zustand (estado) · docx (export Word)

vite 5 · @vitejs/plugin-react · tailwindcss 3 · postcss · autoprefixer

Base de datos

PostgreSQL gestionado (Railway en producción). En local, docker-compose.yml levanta postgres:16-alpine en el puerto 5433 con un volumen persistente y healthcheck. config.py normaliza la URL: si el proveedor inyecta postgres:// sin driver, la reescribe a postgresql+psycopg://.
