Flujo de una sesión, de extremo a extremo

Dónde vive cada dato y qué componente lo mueve, paso a paso:

navegador → API
1 · Login
El usuario inicia sesión: el SPA llama POST /api/v1/auth/login, guarda el JWT y el consultorio activo en authStore, y desde ahí Axios inyecta Authorization: Bearer en cada request.
SPA → wizard
2 · Onboarding
Si el consultorio no está configurado, el guard redirige a /wizard. Los cinco pasos persisten en el backend (procedimientos, costos, tarifas) usando los defaults de la especialidad.
SPA → API → DB
3 · Datos operativos
Carga masiva de facturas (POST /facturas/bulk) y registro de movimientos de caja. El backend valida mínimos (≥20 facturas, ≥3 pagadores, etc.) y persiste en PostgreSQL.
API · servicios
4 · Análisis y pricing
Los reportes (/reportes/analytics, /reportes/pdf) y el módulo de pricing intelligence (/pricing/benchmark, /pricing/simulate) consumen las facturas y tarifas para devolver KPIs y simulaciones.
API → Ollama → DB
5 · Plan con IA
El usuario rellena el contexto y dispara POST /planes. El backend arma un prompt, llama a Ollama, parsea el JSON jerárquico y guarda PlanAccion → Milestones → Fases → Tareas. Cada generación queda en GeneracionIALog.
SPA
6 · Seguimiento
El médico marca tareas como completadas; el progreso cascadea de tareas a fases, milestones y plan. El dashboard refleja el estado actual de los KPIs frente al baseline.

Principio · separar facturación y caja

Las facturas representan derechos de cobro (con su estado); la caja representa el flujo real de dinero. Mantenerlas en módulos distintos —con un puente opcional factura_id en los movimientos— permite ver el desfase entre facturar y cobrar, especialmente con pagadores que demoran.

Principio · IA local y auditable

La generación de planes corre en Ollama dentro de la infraestructura del usuario: los datos sensibles no salen a una API de terceros. Cada llamada queda registrada en GeneracionIALog (prompt, respuesta cruda, tiempo, errores) para depurar y auditar.
