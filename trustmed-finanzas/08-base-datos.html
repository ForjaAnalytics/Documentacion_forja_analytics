08 · Datos & API

Base de datos

24 tablas en PostgreSQL 16 modeladas con SQLAlchemy 2.0. Agrupadas en cinco bloques: identidad y configuración, operación clínica, facturación, caja y planes con IA. Migraciones versionadas con Alembic.

Relaciones principales

User 1 ──< N Consultorio 1 ──< N Configuracion · Procedimiento · Cliente · Doctor · CuentaCaja · …
Procedimiento 1 ──< N ProcedimientoTarifa >── 1 TipoPagador
Factura >── 1 Cliente · TipoPagador · Procedimiento  ·  MovimientoCaja >── 1 Factura? (puente opcional)
PlanContexto 1 ──< N PlanAccion 1 ──< N PlanMilestone 1 ──< N PlanFase 1 ──< N PlanTarea (autorreferencia para subtareas)

Migraciones (Alembic)

El esquema vive en alembic/versions/; nunca se usa create_all. El Procfile ejecuta alembic upgrade head antes de levantar la API en Railway, así cada despliegue aplica los pendientes. Revisiones recientes: campos de contexto del plan, valor_facturado/valor_pagado en facturas, tablas del módulo caja, tabla de doctores y limpieza de porcentaje_copago y margen.

JSON donde tiene sentido

Las partes con forma variable —metricas, resultados, kpis_*, prompt_enviado, respuesta_cruda— se guardan como JSON. Mantiene la flexibilidad para evolucionar prompts y KPIs sin migraciones.

Fallback en local. Si no se define PRODUCTION_DATABASE_URL, config/database.py cae a SQLite (sqlite:///./test.db) para que el dev pueda arrancar sin Docker ni Postgres a mano.
