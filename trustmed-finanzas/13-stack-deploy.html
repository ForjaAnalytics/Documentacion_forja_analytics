Stack & Deploy

Backend y frontend se despliegan por separado: la API en Railway, el SPA en Vercel. Cada uno con su archivo de configuración en el repo y deploy automático en cada push.

Backend Railway (NIXPACKS)

El railway.json + nixpacks.toml definen el build. El Procfile arranca con las migraciones aplicadas antes de levantar Uvicorn:

# Procfile
web: alembic upgrade head && uvicorn app.main:app --host 0.0.0.0 --port $PORT

El healthcheck apunta a /health y la política de reinicio es ON_FAILURE (con tope de reintentos). El JSON de Railway:

// railway.json
{
  "build":  { "builder": "NIXPACKS" },
  "deploy": {
    "startCommand": "alembic upgrade head && uvicorn app.main:app --host 0.0.0.0 --port $PORT",
    "healthcheckPath": "/health",
    "restartPolicyType": "ON_FAILURE"
  }
}

Frontend Vercel (estático)

Build con Vite (npm run build: tsc + vite build). Vercel detecta Vite automáticamente; las variables VITE_* se configuran en el panel del proyecto. El dominio personalizado es finance.visittrustmed.com.

# package.json (scripts)
{
  "scripts": {
    "dev":     "vite",
    "build":   "tsc -b && vite build",
    "preview": "vite preview"
  }
}

Base de datos

PostgreSQL 16 gestionado en Railway. En local, docker-compose.yml levanta un Postgres con datos persistentes; si no hay PRODUCTION_DATABASE_URL configurado, el backend cae a sqlite:///./test.db y permite arrancar sin Docker.

Ollama (IA local)

Servicio aparte: el equipo lo corre en una VM o en la misma máquina del backend. No es parte del despliegue automático de Railway; se gestiona como dependencia operativa (instalar Ollama y descargar el modelo Mistral). Si no está disponible, el endpoint de planes responde 503.
