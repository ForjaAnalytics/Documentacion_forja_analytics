14 · Infraestructura

Operación

Cómo levantar el sistema en local, qué variables necesita y los puntos delicados que conviene conocer antes de operarlo.

Puesta en marcha local

# 1 · Base de datos (Postgres en Docker)
docker compose up -d        # dentro de backend/

# 2 · Backend (FastAPI)
cd backend
python -m venv venv && source venv/bin/activate    # o venv\Scripts\activate en Windows
pip install -r requirements.txt
cp .env.example .env        # y completar las claves
alembic upgrade head        # aplica migraciones
uvicorn app.main:app --reload --port 8000

# 3 · Frontend (Vite en :5173)
cd frontend
npm install
npm run dev

# 4 · Ollama (IA local) — opcional, requerido solo para planes
ollama serve                # http://localhost:11434
ollama pull mistral

El dev del frontend asume que el backend está en http://localhost:8000/api/v1 (ver VITE_LOCALHOST_API_URL). Para usar el backend de producción mientras se desarrolla, basta poner VITE_USE_LOCALHOST=false.

APP_NAME · APP_VERSION · DEBUG · ENVIRONMENTIdentidad y entorno de la app.

VITE_PRODUCTION_API_URL · VITE_LOCALHOST_API_URL frontendURLs base del backend para cada entorno.

Testing

Tests con pytest en backend/tests/. Hay tests unitarios del servicio de auth (9/9 passing). Para correrlos:

cd backend
pytest tests/ -v
pytest --cov=app --cov-report=html      # reporte de cobertura

Migraciones de base de datos

# Crear nueva migración a partir de cambios en los modelos
alembic revision --autogenerate -m "descripcion del cambio"

# Aplicar pendientes
alembic upgrade head

# Volver una revisión
alembic downgrade -1

En producción se aplican solas: el Procfile arranca con alembic upgrade head antes de Uvicorn.

Puntos delicados

Mínimos de carga de facturas

La validación rechaza cargas con menos de 20 facturas, 3 pagadores o 3 meses. Si un usuario reporta que el bulk falla, suele ser por mínimos no cumplidos o por el rango de monto ($1.000 – $50.000.000) que filtra cifras claramente erróneas.

Ollama no es opcional para planes

Sin Ollama corriendo, POST /planes devuelve 503. Si la generación de un plan es lenta o expira, conviene revisar OLLAMA_TIMEOUT y el modelo configurado (modelos más grandes tardan más).

Saldo de caja desincronizado

Los movimientos guardan saldo_anterior y saldo_posterior para auditoría. Si el saldo de una cuenta no cuadra, lo más rápido es ordenar sus movimientos por fecha y verificar continuidad; un cambio manual en la DB pierde la cadena.

Fallback a SQLite

En local, si no hay PRODUCTION_DATABASE_URL se usa SQLite. Comodísimo para arrancar, pero algunas funciones (JSON path queries, performance del bulk) sí difieren de Postgres: para pruebas serias usar Docker.

Observabilidad. Los logs se emiten en formato JSON (python-json-logger) por request y por evento de negocio; el nivel se controla con LOG_LEVEL. El frontend reporta errores 401/5xx al usuario y limpia la sesión cuando es necesario.

Documentación operativa adicional en el propio repo del backend: RAILWAY_DEPLOYMENT.md, DATABASE_MIGRATION_GUIDE.md, OLLAMA_SETUP.md y OLLAMA_INTEGRATION.md.
